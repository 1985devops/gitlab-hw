### Задание 1.

1. `Восстановление за предыдущий день`

* Полное резервное копирование (Full backup). Полная копия базы создается каждую ночь или же как горячее резервное копирование (Hot backup). Если что то случится, восстановится за прошлый день. Требует очень много места на дисках.

2. `Восстанавление за час до предполагаемой поломки.`

* Необходимо использовать комбинированный подход. Каждый час выполняется инкрементальное резервное копирование (Incremental backup), которое сохраняет только блоки данных, измененные с момента последнего задания. Или непрерывное копирование журналов транзакций. При восстановлении, сначала восстанавливается начальная полная копия, а затем накатываются изменения из инкрементальных блоков или журналов транзакций до момента сбоя. 

3. `Моментальное переключение на другую базу`

* Для этого применяется Репликация базы данных (Database Replication). Данные автоматически копируются в реальном времени с основного сервера (master) на один или несколько подчиненных серверов (slave). Если master выходит из строя, то slave берет на себя его роль. Но если допущена была ошибка в базе или удаление, то данные с ошибкой будут на всех серверах. Это помогает только при отказе оборудования или недоступности сети master сервера.


### Задание 2.

1. `Команда резервирования`

* pg_dump -U alex -h localhost -p 5432 -d netology -F c -b -v -f /путь к файлу/mybase1.dump

2. `Восстановление базы данных`

* pg_restore -h localhost -U alex -d netology_restored -v /path/to/backup/mybase1.dump

3. `Автоматизация процесса резервного копирования`

* Делаем скрипт,файл конфигурации и с помощью планировщика заданий, настраиваем задачу в cron по нужному времени бэкапа. Или использовать специальные утилиты разных разработчиков.

### Задание 3.

1. `Пример команды инкрементного резервного копирования базы данных MySQL.`

```
Исходная полная копия.
mysqlbackup --user=root --password=secret --backup-dir=/backup/full backup

Инкрементная копия (добавляет только изменения с момента полной копии)
mysqlbackup --user=root --password=secret \
  --backup-dir=/backup/inc1 \
  --incremental backup

```

2. `Преимущества использования реплики по сравнению с обычным резервным копированием`

* Снижение нагрузки на основной сервер (Master). Резервное копирование (полное или инкрементное) можно запускать на реплике, полностью сняв нагрузку с основного сервера.
* Минимизация времени простоя. Если реплика близка к синхронной, в случае сбоя можно переключить трафик на реплику.
* Снятие нагрузки чтения (Read Scaling). Реплику можно использовать для разгрузки основной базы, направлять на нее запросы на чтение от пользователей. Бэкап так использовать нельзя.
* Бэкап без остановки бизнесса. Можно длать дамп с сервера реплики.
